## What Was Implemented

Two-level Sparse Merkle Tree state database for the Basis Network zkEVM L2, translating
the formally verified TLA+ specification (`StateDatabase.tla`, TLC PASS: 883 states,
5 invariants, 0 violations) into production Go code.

### Architecture

- **Level 1 (Account Trie)**: Maps addresses to `Poseidon(balance, storageRoot)`
- **Level 2 (Storage Tries)**: One SMT per contract, mapping slots to field element values
- **Hash function**: Poseidon2 2-to-1 compression over BN254 scalar field (gnark-crypto v0.20.1)
- **Update algorithm**: O(depth) incremental WalkUp (no full tree rebuild)

### Operations Implemented (matching TLA+ actions)

| TLA+ Action | Go Method | Description |
|-------------|-----------|-------------|
| `CreateAccount(addr)` | `StateDB.CreateAccount(addr)` | Activate dormant account, single WalkUp |
| `Transfer(from, to, amount)` | `StateDB.Transfer(from, to, amount)` | Two-step WalkUp (sender, receiver) |
| `SetStorage(contract, slot, value)` | `StateDB.SetStorage(contract, slot, value)` | Two-level WalkUp (storage, account) |
| `SelfDestruct(contract, beneficiary)` | `StateDB.SelfDestruct(contract, beneficiary)` | Kill + clear storage + two-step WalkUp |
| `GetStorage` (read-only) | `StateDB.GetStorage(contract, slot)` | Verified by StorageIsolation invariant |

## Decisions and Rationale

1. **TreeKey as [32]byte**: Avoids Montgomery form issues with gnark-crypto's fr.Element
   for bit extraction during tree traversal. Only converted to fr.Element for hashing.

2. **Account hash = Poseidon(balance, storageRoot)**: Matches TLA+ spec exactly.
   Nonce and codeHash fields are present in the Account struct (EVM-ready) but omitted
   from the trie hash. The spec notes they are orthogonal to the verified invariants.

3. **Per-SMT permutation instance**: Each SparseMerkleTree owns its Poseidon2 permutation
   for thread-safety isolation. Acceptable memory overhead; can optimize with shared
   instances or sync.Pool when needed.

4. **Struct equality (==) instead of .Equal()**: gnark-crypto's fr.Element methods use
   pointer receivers, preventing calls on non-addressable return values. Since fr.Element
   is [4]uint64 with canonical Montgomery form, struct equality is correct and cleaner.

5. **Sparse node map optimization**: setNode removes entries that equal the default hash
   at that level, keeping memory proportional to non-empty leaves only.

6. **gnark-crypto v0.20.1**: Matches scientist's validated dependency. Required updating
   go directive from 1.22 to 1.25.7 (handled automatically by go mod tidy).
