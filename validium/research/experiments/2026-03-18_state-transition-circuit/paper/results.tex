\subsection{Constraint Scaling Law}

The per-transaction constraint count follows an exact linear relationship
with tree depth:

\begin{equation}
C_{\text{tx}} = 1{,}038 \cdot (d + 1)
\label{eq:constraint-formula}
\end{equation}

This formula has \textbf{zero residual error} across all seven
measurements:

\begin{itemize}
\item $d{=}10$: $1{,}038 \times 11 = 11{,}418$ (measured: 11,418)
\item $d{=}20$: $1{,}038 \times 21 = 21{,}798$ (measured: 21,798)
\item $d{=}32$: $1{,}038 \times 33 = 34{,}254$ (measured: 34,254)
\end{itemize}

The total circuit constraint count is:
\begin{equation}
C_{\text{total}} = 1{,}038 \cdot (d + 1) \cdot n + 3
\label{eq:total-formula}
\end{equation}

where the additive constant of~3 represents the final root equality
check and metadata constraints.

\subsubsection{Reconciliation with Theory}

Comparing the measured formula~(\ref{eq:constraint-formula}) with the
theoretical prediction~(\ref{eq:theory-cost}):

\begin{itemize}
\item \textbf{Predicted}: $C_{\text{tx}} = 1{,}038d + 485$ (linear in $d$)
\item \textbf{Measured}: $C_{\text{tx}} = 1{,}038(d+1) = 1{,}038d + 1{,}038$
\end{itemize}

The per-level coefficient of 1,038 matches exactly, confirming the
$2 \times 519$ structure (two Merkle path verifications, each requiring
519 constraints per level: Poseidon(2) at 240 + 2$\times$Mux1 at
$\sim$139.5 each). The intercept difference (1,038 vs.~485) indicates
that the two leaf Poseidon hashes, the IsEqual check, and associated
routing contribute a full additional ``level'' of constraints, which
the Circom compiler flattens into the $(d+1)$ factor.

\subsection{Depth Scaling}

rather than~$d$. This is significant for capacity planning:
doubling the tree depth from 16 to 32 increases per-transaction

\subsection{Proof Size Invariance}

Groth16 proof size is constant across all configurations: 803--807~bytes.
This is consistent with the theoretical Groth16 proof structure of
3~BN128 group elements (2~$\mathbb{G}_1$ points + 1~$\mathbb{G}_2$ point)
plus formatting overhead. The 4-byte variation is due to JSON
serialization differences in the SnarkJS proof output.
